斜杠命令

将重复的指令封装成一个单词，让每个人都能用同一套标准驱动 Claude — 这是 Claude Code 最容易上手、回报最快的功能。

⏱ 预计 45 分钟
🟢 难度：入门
🔑 MIT 许可
先从一个真实场景说起

假设你每天要让 Claude 帮你 Code Review。你会怎么做？大多数人的做法是每次都打一段类似这样的话：

请审查这段代码，重点检查：
1. 有没有 SQL 注入风险
2. 有没有未处理的异常
3. 命名是否符合我们团队的 camelCase 规范
4. 有没有硬编码的配置项
5. 输出格式：先列问题，再给修复建议

这段话你今天打，明天打，后天还要打。更麻烦的是，你的同事可能有不同版本的说法，导致每个人得到的审查质量参差不齐。

⚠️ 这就是问题所在：你把"如何工作"的知识放在了你的脑子里，而不是项目里。新人不知道，换电脑就忘，团队无法统一。

斜杠命令就是来解决这个问题的。你把上面这段话写进一个文件，以后只需要输入 /review，一切自动搞定。

一个帮你理解的类比
⌨️
像键盘快捷键
Ctrl+C 是"复制"的快捷键。斜杠命令是"一大段 Claude 指令"的快捷键。底层操作没变，只是触发变简单了。
📋
像公司 SOP 手册
好公司把流程写成标准手册，不依赖某个老员工的记忆。斜杠命令就是你和 Claude 协作的 SOP，写进文件，人人可用。
🍳
像厨师的配方卡
米其林厨师不靠记忆做菜，每道菜都有精确配方卡。斜杠命令是你给 Claude 的"配方卡"——每次执行都按照同一套标准来。
底层是怎么运作的？

很多人以为斜杠命令是某种"编程接口"，其实它极其简单 — 本质就是一个 Markdown 文本文件。

当你输入 /review 时，Claude Code 做了以下几件事：

1
查找文件
在 .claude/commands/review.md（项目级）或 ~/.claude/commands/review.md（全局）中查找对应文件。文件名就是命令名，review.md → /review。
2
读取文件内容
把整个 Markdown 文件的内容原封不动地读出来，作为"指令上下文"注入到这次对话中。你在文件里写了什么，Claude 就收到什么。
3
替换变量（如有）
如果文件里用了 $ARGUMENTS，Claude Code 会把你在 /review security 中 "security" 这个参数替换进去。就像函数参数一样。
4
开始执行
Claude 以这个文件内容为指令，结合你当前打开的文件或选中的代码，开始执行任务。整个过程你看不见，但你能感受到结果明显更准确、更一致。

💡 理解这一点很重要：斜杠命令没有任何神秘之处。它就是"帮你预先写好、自动发送给 Claude 的提示词"。越好的命令文件，Claude 给出的结果就越稳定、越专业。
命令放在哪里？两种作用域

斜杠命令有两个存放位置，对应两种使用场景：

位置路径适用范围是否提交 Git
项目级 .claude/commands/ 只在当前项目中生效，打开其他项目时看不到 ✅ 推荐提交，团队共享
全局级 ~/.claude/commands/ 在你电脑上所有项目中都能用 ❌ 属于个人配置

同名命令时，项目级优先于全局级。这意味着你可以在全局放通用命令（如 /translate），在项目里放定制化命令（如符合本项目规范的 /review），两者互不冲突。

✅ 团队协作建议：把 .claude/commands/ 目录提交到 Git。这样新同事 clone 项目后，立刻就能用你们团队积累的所有命令，零成本传承经验。
动手创建第一个命令（10 分钟）

我们从零开始创建一个 /review 命令。不需要任何配置，不需要重启，文件一保存立刻生效。

1
创建目录（如果不存在）
# 在你的项目根目录下执行
mkdir -p .claude/commands
2
创建命令文件
新建文件 .claude/commands/review.md，写入以下内容。注意：文件内容越具体，Claude 的输出越稳定。
3
在 Claude Code 中使用
打开 Claude Code，导航到你的项目目录，在对话框输入 /review 即可。如果想重点审查某个方面，输入 /review 安全漏洞，"安全漏洞"就会替换 $ARGUMENTS。

一份高质量命令文件长什么样

下面是一个完整的 review.md 示例。注意观察它的结构：角色定义 → 审查维度 → 输出格式。

# 代码审查命令

## 你的角色
你是一名有 10 年经验的高级工程师，正在对 Pull Request 进行正式 Code Review。
$ARGUMENTS 是需要重点关注的方向（如"安全"、"性能"），如果没有则全面审查。

## 审查维度（按优先级排序）

### 🔴 必须修复（阻塞合并）
- **安全漏洞**：SQL 注入、XSS、CSRF、未校验的用户输入、硬编码密钥
- **逻辑错误**：边界条件未处理、空指针风险、并发问题
- **数据完整性**：事务缺失、异常时数据不一致

### 🟡 建议修复（不阻塞但应尽快处理）
- **性能问题**：N+1 查询、不必要的全表扫描、循环中的 I/O 操作
- **错误处理**：未捕获的异常、错误信息暴露给用户

### 🟢 优化建议（可以放到下个迭代）
- **可读性**：命名不清晰、函数过长（超过 50 行）、魔法数字
- **规范符合度**：是否符合 camelCase 命名、是否有必要的注释

## 输出格式
请严格按以下 Markdown 表格输出，不要输出其他内容：

| 行号 | 问题描述 | 严重程度 | 修复建议 |
|------|---------|---------|---------|
| ... | ... | 🔴/🟡/🟢 | ... |

表格之后，附上"优化后的完整代码"。

同样让 Claude 审查一段代码，看看两种方式的输出质量差异：

你输入："帮我看看这段代码"

Claude 输出：笼统的段落式建议，格式每次不一样，不知道严重程度，也不知道哪行有问题，需要反复追问。

你输入："/review"

Claude 输出：标准化表格，按严重程度分级，精确到行号，附带修复代码。每次格式完全一致，可直接复制进 PR 评论。

⏱ 30 秒得到可用结果，无需追问

变量系统：让一个命令应对多种场景

命令文件里可以使用以下变量，让同一个命令变得更灵活：

变量含义使用示例替换后的值
$ARGUMENTS 斜杠命令后面跟的参数 输入 /review 安全 安全
$FILE 当前在编辑器中打开的文件路径 文件里写 审查 $FILE src/auth/login.ts
$SELECTION 你在编辑器中选中的代码片段 文件里写 解释这段代码：$SELECTION 选中的代码文本

灵活使用 $ARGUMENTS 的技巧

在命令文件里，你可以这样写，让 Claude 根据参数自动调整行为：

# /fix 命令文件

请修复代码中的问题。

$ARGUMENTS 如果有值，则只修复 $ARGUMENTS 相关的问题。
$ARGUMENTS 如果为空，则修复所有能发现的问题。

修复完成后，输出：
1. 修改了什么（一句话）
2. 为什么这样改
3. 修改后的完整代码

这样，/fix 修复所有问题，/fix 类型错误 只修复类型相关的问题。一个命令文件服务多种需求。

推荐立刻创建的 6 个命令

以下是最常用的命令，按"立刻能产生价值"排序，建议按顺序逐个创建：

🔍
/review
代码审查。最高优先级创建，能立刻统一团队审查标准，消除"每次问法不一样"的问题。
🧪
/test
生成单元测试。指定测试框架（Jest/Vitest/PyTest）、覆盖要求（边界条件/异常路径），让测试质量稳定。
📝
/docs
生成文档注释。指定中英文、JSDoc/Docstring 格式，让所有函数注释风格统一。
🐛
/debug
错误分析。传入错误信息和堆栈，按"根因→原因链→修复方案"格式输出，避免 Claude 给出"可能是..."式的模糊回答。
♻️
/refactor
代码重构。指定遵循的原则（SOLID/DRY/KISS）和约束（不改变接口、不引入新依赖），避免 Claude 过度重构。
📊
/explain
代码讲解。指定受众（初级/高级工程师），让 Claude 调整解释的深度，避免过于简单或过于复杂。
3 个常见误区，避免踩坑
!
误区一：命令文件越短越好
错误想法：命令文件就写一行"请审查代码"，简洁。
真相：写得越具体，输出越稳定。模糊的指令会让 Claude 每次"发挥"不同，反而需要你更多追问。把你脑子里的判断标准全写进去。
!
误区二：命令文件是一次性的
错误做法：写完就不管了。
正确做法：把命令文件当作活文档维护。每当你发现 Claude 给出的结果不对，就回去更新命令文件。团队成员发现好的改进也可以提 PR 更新。
!
误区三：全局命令和项目命令混淆
错误做法：把所有命令都放全局。
正确做法：通用命令（translate、explain）放全局；项目专属命令（符合本项目规范的 review、符合本项目技术栈的 test）放项目。同名时项目命令会覆盖全局命令。

记忆管理

Claude 默认是"失忆的"。通过 CLAUDE.md 给它注入持久记忆，让它真正理解你的项目，而不是每次都从零开始。

⏱ 预计 40 分钟
🟢 难度：入门
Claude 天生是个"失忆症患者"

你有没有遇到过这种情况：昨天刚告诉 Claude 你们项目用的是 TypeScript + Fastify，今天打开新会话，它又给你写了 Express + JavaScript 的代码？

或者你明明强调过"错误处理要用我们统一的 AppError 类"，但下次生成代码时它又直接 throw new Error()？

⚠️ 这不是 Bug，这是设计。 Claude 的每一次会话都是全新的对话，它不会记得上一次你说过什么。如果你不告诉它，它就只能用"猜测"来填补空白。

更麻烦的是，每次都要重新解释项目背景，既浪费时间，又容易遗漏细节，导致生成的代码"看起来对，但不符合你们项目规范"。

CLAUDE.md 就是解法 — 一个专门让 Claude 在会话开始时自动读取的文件，把项目知识永久写进去，一劳永逸。

怎么理解 CLAUDE.md？
📖
新员工入职手册
好公司会给新员工一本手册：公司用什么工具、代码规范是什么、哪些是禁区。CLAUDE.md 就是你给 Claude 写的这本手册，它每次"上班"都会先读一遍。
🧭
导航 App 的地图数据
导航 App 不靠"记忆"，靠的是提前下载好的地图。CLAUDE.md 是你项目的"地图"，Claude 每次都能准确导航，而不是靠直觉猜方向。
🎬
电影拍摄的剧本
导演不会每天早上给演员重新解释剧情，大家拿着同一份剧本就对齐了。CLAUDE.md 是你和 Claude 协作的"剧本"，不用每次重新对焦。

Claude Code 支持三个不同位置的 CLAUDE.md，它们同时生效，形成"由粗到细"的记忆层级：

层级位置写什么举例
全局 ~/.claude/CLAUDE.md 跨项目的个人偏好 用中文回复、偏好函数式风格、不要写过度注释

Claude 是怎么读取它的？

很多人以为 CLAUDE.md 是某种"数据库"或"配置文件"。其实它的机制非常简单：

1
会话启动时自动扫描
当你在项目目录里打开 Claude Code 时，它会自动扫描三个位置的 CLAUDE.md 文件，不需要任何手动操作。
2
内容注入"系统提示词"
找到文件后，Claude Code 把所有 CLAUDE.md 的内容合并，作为系统提示词（System Prompt）的一部分注入到会话中。这就是为什么 Claude 一开口就"知道"你的项目规范。
3
整个会话持续有效
注入后，这些内容在整个会话期间都有效。你不需要重复强调，Claude 会在每次回答时自动参考这些规则。
⚠️ 一个重要限制：CLAUDE.md 的内容会占用上下文窗口（Context Window）。写太多会压缩 Claude 实际工作的空间。原则：写关键约束，不写教程。每个项目的 CLAUDE.md 建议控制在 200 行以内。
有没有 CLAUDE.md：两种体验对比
❌ 没有 CLAUDE.md
  • 你说"给我生成一个用户服务"，Claude 生成了 Express + MongoDB 的代码（你们用的是 Fastify + PostgreSQL）
  • 错误处理直接 throw Error，不符合你们的 AppError 规范
  • 函数命名用了 snake_case，你们约定的是 camelCase
  • 代码放在了错误的目录层
  • 每次开新会话，要花 5 分钟重新解释项目背景
✅ 有 CLAUDE.md
  • 生成的代码直接用 Fastify + PostgreSQL，符合技术栈
  • 错误处理自动使用 AppError 类
  • 命名规范、目录位置全部正确
  • 不用反复纠正，一次到位
  • 新会话 0 成本，打开即用
一份高质量 CLAUDE.md 的完整模板

好的 CLAUDE.md 不是流水账，而是精准的约束声明。下面是一个经过实践验证的结构：

# 项目：[项目名称]
# 最后更新：[日期]

## 技术栈
- 后端：Node.js 20 + Fastify 4 + Prisma 5
- 前端：React 18 + TypeScript 5 + Tailwind CSS 3
- 数据库：PostgreSQL 16（主库）+ Redis 7（缓存/队列）
- 测试：Vitest + Testing Library
- 部署：Docker + GitHub Actions → AWS ECS

## 目录结构与职责
src/
  api/          # 路由层 ← 只做参数校验和响应格式化，禁止写业务逻辑
  services/     # 业务逻辑层 ← 所有业务判断都在这里
  repositories/ # 数据层 ← 封装所有 Prisma 操作，services 不直接用 prisma
  utils/        # 纯工具函数 ← 无副作用，可独立测试
  types/        # TypeScript 类型定义

## 代码规范（必须遵守）
- 命名：camelCase（变量/函数）、PascalCase（类/类型）、kebab-case（文件名）
- 注释：中文，只注释"为什么"，不注释"是什么"
- 函数长度：单个函数不超过 40 行，超过则拆分
- 导入顺序：Node 内置 → 第三方 → 内部模块（空行分隔）

## 错误处理规范
- 统一使用 AppError 类（在 utils/errors.ts），禁止直接 throw new Error()
- AppError 构造：new AppError('USER_NOT_FOUND', '用户不存在', 404)
- 所有 async 函数必须有 try/catch 或上层统一处理

## 禁止事项（会被 Code Review 打回）
- 禁止在 api 层直接调用 prisma
- 禁止硬编码任何配置值，统一从 config/index.ts 读取
- 禁止 console.log 进生产代码，使用 logger 模块
- 禁止在循环内执行数据库查询（N+1 问题）

- 正在开发：用户权限系统（RBAC）

3 个常见误区
!
误区一：写得越详细越好
错误：把整个项目 Wiki 都粘进去，洋洋洒洒几千行。
真相：CLAUDE.md 会占用上下文窗口。内容越多，留给 Claude 实际工作的空间越少，而且过多信息反而会被稀释。建议控制在 200 行以内，聚焦约束而非教程。
!
误区二：写完就不更新
错误：项目初始化时写一次，之后从不维护。
真相：CLAUDE.md 是活文档，应该随项目演进。每次发现 Claude 重复犯同一类错误，就把对应规则加进去。把"修复 Claude 错误"变成"更新 CLAUDE.md"的习惯。
!
误区三：一个文件写所有规则
错误：把前端、后端、数据库的所有规则都塞进根目录的 CLAUDE.md。
真相：利用目录级 CLAUDE.md。比如在 src/api/CLAUDE.md 里单独写接口层规范，在 src/services/CLAUDE.md 里写业务层规范，相关规则只在相关目录生效，精准而不噪杂。
📌 本模块核心要点
  • Claude 每次会话都是全新的，CLAUDE.md 是解决"失忆"问题的唯一持久化手段
  • 三层体系：全局（个人偏好）→ 项目（团队规范）→ 目录（模块规则），越精细优先级越高
  • 内容原则：写约束和禁止事项，不写教程；控制在 200 行以内
  • 用 ./CLAUDE.md 提交 Git，让团队共享；不要把项目 Wiki 全粘进去
  • 当 Claude 反复犯同一类错误时，就是该更新 CLAUDE.md 的信号

技能 Skills

CLAUDE.md 装的是"项目规范"，Skills 装的是"专业能力"。按需加载，用完释放，让 Claude 在需要时变成某个领域的专家。

⏱ 预计 60 分钟
🟡 难度：进阶
一个新的问题：上下文被撑爆了

学完 CLAUDE.md 你可能会想：那我把所有知识都写进 CLAUDE.md 不就行了？安全审查的规则、性能优化的技巧、架构设计的原则……全写进去！

这个想法听起来合理，但有个致命问题：Claude 的上下文窗口是有限的。

💡 上下文窗口是什么？你可以把它理解成 Claude 的"工作桌"——桌面就这么大，CLAUDE.md、当前代码、对话历史、你的问题……都要放在这张桌上。桌上的东西越多，Claude 能真正"专注处理"的内容就越少。

如果你把所有知识都塞进 CLAUDE.md，会发生什么？

  • 每次对话都要消耗大量 token 加载这些内容，大部分时候根本用不上
  • 上下文被占满，Claude 处理实际代码的"空间"变小，质量下降
  • 维护难度极高，一个文件几千行，改都不敢改

Skills（技能）就是这个问题的解法：把专业知识模块化，只在需要的时候加载，用完释放，随用随取。

一个直觉类比
👨‍⚕️
医生查阅专科手册
全科医生不把所有专科知识都背在脑子里。需要心脏专科建议时，查心脏手册；需要骨科建议时，查骨科手册。Skills 就是这些专科手册。
🔌
按需加载的插件
你不会同时开启所有浏览器插件，需要哪个激活哪个。Skills 让 Claude 在需要时"激活"专业能力，不需要时不占资源。
📚
图书馆的专业书架
你不会把图书馆所有书都搬回家，需要时去图书馆借。Skills 是 Claude 的专业书架，用完放回去，下次再需要时再借。

# 技能：安全审查专家

## 你的角色
你是一名专注于 Web 应用安全的高级安全工程师，
拥有 OWASP Top 10 的深度理解和多年安全审计经验。
当这个技能被激活时，你要用安全视角审查每一行代码。

## 审查框架（按威胁严重程度）

### P0 — 立即阻断（不修复不允许上线）
- SQL 注入：用户输入直接拼接到 SQL 语句
- 命令注入：未验证的输入传入 exec/spawn
- 认证绕过：JWT 验证缺失或签名未校验

### P1 — 高优先级（本次 Sprint 必须修复）
- XSS：用户输入未转义直接渲染到 HTML
- CSRF：状态变更接口缺少 Token 校验
- 敏感信息泄露：密码/密钥出现在日志、响应体、错误信息中

### P2 — 中优先级（下个 Sprint 修复）
- 不安全的直接对象引用（IDOR）：通过 ID 访问他人资源
- 权限校验遗漏：只验证了登录状态，没验证操作权限
- 不安全的反序列化：直接 JSON.parse 不受信任的数据

| 位置 | 漏洞类型 | 严重程度 | 问题描述 | 修复代码 |

| 行号 | 类型 | P0/P1/P2 | 具体说明 | 修复后的代码片段 |

子代理 Subagents

一个 Claude 搞不定的事，就让多个 Claude 分工合作。子代理是突破上下文限制、处理大型复杂任务的核心武器。

⏱ 预计 75 分钟
🟡 难度：进阶
先说一个真实的瓶颈

假设你要用 Claude 完成这样一个任务：把一个 10 万行的老项目从 JavaScript 重构为 TypeScript。

你打开 Claude Code，把项目目录给它，然后说"帮我全部重构成 TypeScript"。

然后……它卡住了，或者只处理了一小部分就停下来，或者后面的代码开始"忘记"前面定的类型规范。

⚠️ 这是上下文窗口的物理极限。Claude 的"工作记忆"是有限的。一个会话能处理的代码量、对话历史、指令加在一起，超过上限就会开始丢失信息。这不是 Claude 不够聪明，是物理约束。

子代理（Subagents）就是为了打破这个约束而生的。核心思路是：把大任务拆成独立的小任务，每个小任务由一个独立的 Claude 实例处理，最后由主代理汇总结果。

一个帮你理解的类比
👔
项目经理 + 专项小组
一个人不可能同时深度思考前端架构和数据库设计。项目经理把任务分给前端组、后端组、测试组，各自独立工作，最后 PM 汇总。主代理就是 PM，子代理就是各专项小组。
🏭
流水线分工
汽车工厂不是一个工人从零装好一辆车，而是多条流水线并行，每条线做一个零部件，最后组装。子代理就是各条流水线，互不干扰，效率翻倍。
🎭
导演 + 演员团队
导演（主代理）负责整体叙事和调度，演员（子代理）各自专注自己的角色，互不越界。导演不亲自表演每一场戏，但掌控全局节奏。
子代理的底层机制

很多人以为子代理是某种特殊 API。其实它的原理非常简单：

1
每个子代理是独立的 Claude 会话
子代理有自己独立的 messages[] 上下文，与主代理和其他子代理完全隔离。这意味着每个子代理的"工作桌"都是干净的，不会被其他任务的上下文污染，也不会撑爆。
2
主代理负责任务拆分和信息传递
主代理分析大任务，决定怎么拆分，然后把精确的子任务描述 + 必要的上下文发给每个子代理。子代理之间不直接通信，所有协调都通过主代理。
3
结果汇总
各子代理完成任务后，把结果返回给主代理。主代理负责整合、处理冲突、输出最终结果。整合阶段可能需要主代理做一些"翻译"和"缝合"工作。

💡 关键理解：子代理之间不共享上下文。子代理 A 不知道子代理 B 在做什么，除非主代理明确把信息传递过去。这既是优点（防止干扰），也是需要注意的限制。
单代理 vs 子代理：什么时候该用哪个？
场景特征用单代理用子代理
任务规模单文件或少量文件多模块、多文件、跨目录
任务之间的关联强依赖，需要上下文连贯相互独立，可以并行
上下文需求需要完整历史记录每个子任务只需少量上下文
时间要求对速度不敏感希望并行加速
典型例子修复单个 Bug、解释一段代码重构整个模块、批量生成测试
✅ 判断标准：如果你的任务可以被描述为"同时做 A、同时做 B、同时做 C，最后合并"，就该用子代理。如果必须"先做 A，再根据 A 的结果做 B"，则保持单代理。
有无子代理：体验对比
❌ 单代理处理大任务
  • 处理到一半上下文满了，后面的内容开始"遗忘"
  • 前面定好的类型规范，后面的代码开始不遵守
  • 只能串行处理，一个文件一个文件来，很慢
  • 出错了很难定位，所有内容混在一个对话里
✅ 子代理并行处理
  • 每个子代理上下文干净，专注一件事，质量稳定
  • 多个子代理同时工作，总时间大幅缩短
  • 某个子代理出错，不影响其他子代理
  • 每个子任务结果独立可验证，便于调试
如何触发子代理？提示词写法

子代理不需要特殊语法，关键是在提示词里明确声明任务拆分结构，让 Claude 理解这是一个需要多实例协作的任务。

写法一：明确声明子代理角色

请使用子代理模式完成以下任务：

主代理负责：协调各子代理的工作，最终整合结果。

子代理 A（前端专家）负责：
- 分析 src/components/ 下的所有组件
- 为每个组件生成对应的 TypeScript 类型定义
- 确保 Props 类型完整且准确

子代理 B（后端专家）负责：
- 分析 src/api/ 下的所有接口
- 生成对应的请求/响应类型
- 处理错误类型定义

子代理 C（集成专家）负责：
- 基于 A 和 B 的结果生成统一的 types/index.ts
- 确保前后端类型一致，消除重复定义

请按以上分工执行，最后汇总输出完整的类型系统。

写法二：任务清单式（适合批量处理）

请为以下 5 个模块分别启动独立的子代理，每个子代理只处理自己的模块：

1. 子代理→处理 src/user/：生成完整单元测试，覆盖率 > 80%
2. 子代理→处理 src/order/：生成完整单元测试，覆盖率 > 80%
3. 子代理→处理 src/payment/：生成完整单元测试，覆盖率 > 80%
4. 子代理→处理 src/notification/：生成完整单元测试，覆盖率 > 80%
5. 子代理→处理 src/report/：生成完整单元测试，覆盖率 > 80%

每个子代理完成后输出：模块名 + 测试文件内容 + 覆盖率估算。
主代理最后汇总：哪些模块测试通过，哪些有问题需要人工介入。

写法三：流水线式（上下游有依赖）

请按以下流水线执行，每个阶段用独立子代理：

阶段1 子代理（分析师）：
  - 读取 API 文档 docs/api.md
  - 输出：接口清单（JSON 格式）

阶段2 子代理（开发者）：
  - 输入：阶段1 的接口清单
  - 输出：每个接口的 TypeScript 封装函数

阶段3 子代理（测试工程师）：
  - 输入：阶段2 的封装函数
  - 输出：每个函数的单元测试

主代理最后把三个阶段的产物合并成完整 SDK。

注意：每个阶段的子代理只能看到自己阶段的输入，不要携带其他阶段的历史。
子代理的信息传递：最容易出错的地方

子代理最常见的失败原因，不是任务本身，而是信息传递断链。因为子代理之间不共享上下文，主代理必须显式传递关键信息。

场景❌ 错误做法✅ 正确做法
子代理 B 需要用子代理 A 的结果 直接让 B 开始，以为它"知道" A 做了什么 主代理把 A 的输出摘要明确写进 B 的任务描述
多个子代理需要遵守同一规范 只在主代理提示词里说一次规范 每个子代理的任务描述里都重复一遍关键规范
子代理出错需要重试 重新发起整个任务 只重启出错的那个子代理，其他子代理结果保留
⚠️ 黄金原则：凡是子代理执行时需要知道的信息，一定要在它的任务描述里显式写出来。不要假设子代理能"猜到"或"推断出"上下文。

全栈功能开发

📚
文档批量生成
每个子代理负责一个模块的文档。统一格式在任务描述里写清楚，所有文档风格自动一致。
🔍

3 个常见误区
!
误区一：什么任务都用子代理
错误：修复一个 Bug 也要启动子代理，"看起来更专业"。
真相：子代理有额外的协调成本。小任务用单代理更快，子代理适合的是真正"可并行、体量大"的任务。杀鸡用牛刀反而慢。
!
误区二：以为子代理之间能自动同步
错误：让子代理 A 和 B 同时修改同一个文件，以为它们会自动协调。
真相：子代理之间完全独立，会产生写入冲突。同一个文件同一时刻只能有一个子代理处理，或者让主代理在合并阶段处理冲突。
!
误区三：子任务描述太模糊
错误：给子代理说"处理用户模块"，没有明确边界和输出格式。
真相：子代理的任务描述必须边界清晰、输入明确、输出格式规定好。越模糊的描述，越容易让子代理"自由发挥"，导致结果无法整合。

钩子 Hooks

在 Claude 的每个动作前后插入你的逻辑，把"Claude 做完就结束"变成"Claude 做完自动触发下一步"，实现真正的自动化闭环。

⏱ 预计 60 分钟
🟠 难度：进阶+
没有 Hooks 时，工作流是断的

Claude 生成了代码，写入了文件。然后呢？

没有 Hooks 的世界里，Claude 写完代码就"交差"了。你还需要手动：跑一遍 Prettier 格式化 → 跑一遍 ESLint 检查 → 跑一遍单元测试 → 如果测试失败再回来改 → 改完再手动提交……

⚠️ 这些步骤都是可以自动化的，但没有 Hooks 就没法自动化。Claude 不知道"写完代码"之后还应该做什么，除非你明确告诉它，或者用 Hooks 在系统层面强制执行。

Hooks 就是 Claude Code 提供的"事件监听器"。当 Claude 执行某个动作时，你预先注册的脚本会自动触发，形成完整的自动化闭环。

一个帮你理解的类比
🏭
工厂质检门禁
产品离开生产线时必须经过质检（PostToolUse），不合格的不让出门；进入危险区域时必须先登记（PreToolUse）。Hooks 是代码流水线上的自动质检门禁。
🔔
数据库触发器
数据库里的 TRIGGER：INSERT 后自动更新统计表，DELETE 前自动备份。Hooks 就是 Claude Code 的"触发器"，特定事件发生时自动执行你的脚本。
⚙️
Git Hooks
pre-commit 钩子在提交前自动运行 lint，post-merge 后自动安装依赖。Claude Code 的 Hooks 和 Git Hooks 概念完全一样，只是触发时机是 Claude 的动作。
四个触发时机，各有用途

Claude Code 提供 4 个 Hook 触发点，覆盖了 Claude 工作流的关键节点：

Hook 类型触发时机最适合做什么典型用途
PreToolUse Claude 准备执行某工具前 检查、拦截危险操作 拦截 rm -rf、DROP TABLE，记录操作审计日志
PostToolUse Claude 完成工具调用后 自动化质量检查 写文件后自动格式化、运行测试、检查类型

Hooks 配置在哪里？怎么写？

Hooks 配置在 settings.json 文件里，支持两个位置：

  • ~/.claude/settings.json：全局生效，所有项目都有
  • ./.claude/settings.json：项目级，只在当前项目生效（优先级更高）

基本结构如下，每个 Hook 包含三个要素：触发类型 → 匹配规则 → 执行命令：

{
  "hooks": {
    "PostToolUse": [        // Hook 类型
      {
        "matcher": "Write", // 匹配规则：只在 Write 工具后触发
        "hooks": [{
          "type": "command",
          "command": "npx prettier --write "$CLAUDE_TOOL_INPUT_path""
        }]
      }
    ],
    "PreToolUse": [
      {
        "matcher": "Bash",  // 匹配规则：所有 Bash 命令前触发
        "hooks": [{
          "type": "command",
          "command": "bash ~/.claude/hooks/check-dangerous.sh"
        }]
      }
    ]
  }
}
matcher 匹配规则详解

matcher 决定 Hook 在什么情况下触发，支持精确匹配和通配符：

matcher 写法触发条件适用场景
"Write"Claude 写入文件时写完自动格式化
"Bash"Claude 执行任意 Bash 命令时命令审计、危险命令拦截
"Read"Claude 读取文件时访问日志
"Edit"Claude 编辑文件时（局部修改）编辑后自动 lint
".*" 或不填所有工具调用全量审计日志
4 个可以直接使用的实战配置

① 写文件后自动格式化 + Lint（最常用）

{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Write",
      "hooks": [{
        "type": "command",
        "command": "FILE="$CLAUDE_TOOL_INPUT_path" && npx prettier --write "$FILE" 2>/dev/null; npx eslint --fix "$FILE" 2>/dev/null; echo "✅ 格式化完成: $FILE""
      }]
    }]
  }
}

② 危险命令拦截（安全必备）

# 先创建拦截脚本 ~/.claude/hooks/guard.sh
#!/bin/bash
CMD="$CLAUDE_TOOL_INPUT_command"

# 检测危险命令
if echo "$CMD" | grep -qE "rm -rf|DROP TABLE|DELETE FROM.*WHERE 1|format [A-Z]:"; then
  echo "🚫 危险命令被拦截：$CMD" >&2
  echo "请人工确认后手动执行" >&2
  exit 1  # 退出码非0 = 阻止 Claude 执行
fi

exit 0  # 放行
// settings.json 中启用拦截
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Bash",
      "hooks": [{
        "type": "command",
        "command": "bash ~/.claude/hooks/guard.sh"
      }]
    }]
  }
}

③ 任务完成后发钉钉通知

{
  "hooks": {
    "Stop": [{
      "hooks": [{
        "type": "command",
        "command": "curl -s -X POST 'https://oapi.dingtalk.com/robot/send?access_token=YOUR_TOKEN' -H 'Content-Type: application/json' -d '{"msgtype":"text","text":{"content":"✅ Claude 任务完成！请查看结果。"}}'  > /dev/null"
      }]
    }]
  }
}

④ 全量操作审计日志

{
  "hooks": {
    "PostToolUse": [{
      "hooks": [{
        "type": "command",
        "command": "echo "[$(date '+%Y-%m-%d %H:%M:%S')] Tool: $CLAUDE_TOOL_NAME | Input: $CLAUDE_TOOL_INPUT_path" >> ~/.claude/audit.log"
      }]
    }]
  }
}
✅ 推荐组合：同时启用"自动格式化"+ "危险命令拦截"+ "任务完成通知"，三个 Hook 覆盖了最核心的自动化需求，部署成本低、收益高。
Hook 脚本的退出码很重要

PreToolUse 的 Hook 脚本，退出码决定了 Claude 的行为：

退出码含义Claude 的行为
0检查通过，放行继续执行原来的工具调用
非 0（如 1）检查不通过，拦截取消执行，并把脚本的 stderr 输出给用户
💡 这非常有用：你可以在拦截脚本里用 echo "原因" >&2 输出原因，Claude 会把这段文字展示给用户，解释为什么操作被阻止。
3 个常见误区
!
误区一：Hook 脚本报错会卡住整个流程
现象：格式化脚本因为某个文件类型不支持而报错，导致 Claude 工作流卡住。
解法：PostToolUse 的 Hook 脚本末尾加 || true 或 exit 0，确保即使脚本失败也不影响 Claude 继续工作。PreToolUse 的拦截脚本才应该用非0退出码。
!
误区二：在 PostToolUse 里做耗时操作
现象：每次写文件后都跑完整测试套件，Claude 每写一个文件就要等几分钟。
解法：PostToolUse 只做快速操作（格式化、lint、类型检查）。完整测试放在 Stop Hook 里，或者用后台任务模式异步运行。
!
误区三：忘记处理环境变量
现象：Hook 脚本在命令行手动跑没问题，但在 Claude Code 里触发时找不到 node、npm 等命令。
解法：在 Hook 脚本里显式指定完整路径，如 /usr/local/bin/npx，或者在 settings.json 的 env 字段里设置 PATH。
📌 本模块核心要点
  • Hooks 是 Claude Code 的"事件触发器"，让 Claude 的动作与你的脚本自动联动
  • 4个触发时机：PreToolUse（执行前拦截）/ PostToolUse（执行后处理）/ Notification（状态通知）/ Stop（任务完成）
  • PreToolUse 用非0退出码可以阻止 Claude 执行，是安全门禁的核心机制
  • PostToolUse 只做快速操作，耗时任务放 Stop Hook 或异步执行
  • 优先部署：写文件后自动格式化 + 危险命令拦截，立刻见效

MCP 服务器

Claude 默认只能看代码、写代码。MCP 让它能直接操作你的数据库、读写 GitHub、查询 API——真正打通 Claude 和你的工具链。

⏱ 预计 90 分钟
🔴 难度：高级
Claude 的能力边界在哪里？

没有 MCP 时，Claude Code 能做的事是有边界的：读写本地文件、运行 Shell 命令。这已经很强大，但有一类需求它做不到：

  • 你问它"数据库里用户表有多少条数据"——它不知道，因为它连不上你的数据库
  • 你让它"把这个 Bug 修完自动创建 GitHub Issue"——它做不到，因为它没有 GitHub 访问权限
  • 你想让它"查一下线上 API 的响应时间"——它做不到，因为它没有监控系统的访问能力
⚠️ 根本原因：Claude 是一个 AI 模型，它的能力边界由"工具（Tools）"决定。默认工具只有文件系统操作和 Bash。MCP 就是扩展这个工具集的标准化协议。

MCP（Model Context Protocol）是 Anthropic 制定的开放协议。任何系统都可以按这个协议开发一个"MCP Server"，把自己的能力暴露给 Claude。接入之后，Claude 就能像使用文件系统一样使用这些外部系统。

MCP 的核心概念：两个角色
🤖
MCP Client（Claude Code）
发起请求的一方。Claude 需要调用某个外部能力时，向 MCP Server 发送标准化请求。Claude Code 内置了 MCP Client 功能，不需要你开发。
🔌
MCP Server（你配置的）
提供能力的一方。可以是官方提供的（如 server-github），也可以是你自己写的。每个 Server 暴露一组"工具函数"，Claude 可以像调用本地函数一样调用它们。

列出风险点"

# Claude 会：查数据库表结构 → 扫描代码 → 交叉比对 → 输出风险清单
3 个常见误区
!
误区一：直接把生产数据库连上去
风险：Claude 在分析时可能会执行写操作（如尝试优化、清理数据），直连生产库有数据损坏风险。
正确做法：先连只读副本（Read Replica）或测试环境数据库。确认工作流安全后再考虑生产环境，或者通过 settings.json 的 permissions 限制 Bash 里的数据库写操作。
!
误区二：把 Token 硬编码在 settings.json 里提交 Git
风险：GitHub Token、数据库密码出现在代码仓库里，极其危险。
正确做法：用环境变量引用，在 settings.json 里写 "GITHUB_TOKEN": "$GITHUB_TOKEN"，实际值在系统环境变量或 .env（加入 .gitignore）里配置。
!
误区三：MCP Server 启动失败时不知道怎么排查
现象：配置了 mcpServers 但 Claude 说"无法访问数据库"。
排查步骤：① 在终端手动运行 Server 的启动命令，看有没有报错；② 检查环境变量是否正确设置；③ 检查网络连通性（数据库/API 是否可达）；④ 查看 Claude Code 的日志输出（通常在 ~/.claude/logs/）。
📌 本模块核心要点
  • MCP 是 Claude 扩展能力的标准协议，让它能访问数据库、GitHub、Slack 等任意外部系统
  • 配置简单：在 settings.json 的 mcpServers 字段写入启动命令和认证信息即可
  • 官方提供了 postgres、github、slack 等常用 Server，开箱即用
  • 安全第一：先连只读/测试环境，Token 不要提交 Git
  • 接入 GitHub + 数据库后，Claude 能完成从分析 → 修复 → 提 PR 的全链路自动化

插件系统

MCP 连接外部系统，Plugin 扩展 Claude Code 本身。两者解决不同问题，学会区分才能用对工具。

⏱ 预计 50 分钟
🔴 难度：高级
Plugin 和 MCP 到底有什么区别？

很多人学到这里会困惑：上一个模块刚学了 MCP，这个模块又来了 Plugin，它们都是"扩展能力"的，区别在哪？

💡 一句话区分：MCP 是"给 Claude 接外部工具"；Plugin 是"改造 Claude Code 这个应用本身"。

更直观的类比：

🔌
MCP = 给电脑接外设
就像给电脑接上打印机、摄像头——电脑本身没变，只是多了新的外接能力。MCP 给 Claude 接上数据库、GitHub，Claude 本身没变。
🧩
Plugin = 给 Chrome 装扩展
Chrome 扩展改变的是浏览器本身的行为——加按钮、改界面、拦截请求。Plugin 改变的是 Claude Code 这个工具的行为方式。
对比维度MCP ServerPlugin
改变什么Claude 能访问什么外部资源Claude Code 工具本身的行为
通信方式进程间通信（JSON-RPC over stdio）进程内直接调用（JavaScript）
开发语言任意语言（Python/Go/Node 均可）仅 JavaScript/TypeScript
性能有跨进程通信延迟极低延迟，直接调用
适合做什么连数据库、连 API、连外部服务自定义工具函数、拦截处理、内部集成
典型场景接入 GitHub、Postgres、Slack集成公司内网系统、自定义输出处理
Plugin 能做什么？

Plugin 让你用 JavaScript 编写"工具函数"，Claude 可以在执行任务时调用这些函数，就像调用内置工具一样。

🏢
接入公司内网系统
内网 Jira、内网 Wiki、内网 CI/CD——这些系统没有公开的 MCP Server，你可以自己写 Plugin 接入。
🔐
安全获取密钥
从公司的 Vault、AWS Secrets Manager 动态获取 API Key，不需要在配置文件里硬编码敏感信息。
⚙️
自定义工作流步骤
把公司特有的发布流程、代码检查流程包装成工具函数，Claude 调用时自动执行标准流程。
📊
自定义输出处理
把 Claude 生成的代码自动转换成特定格式，或者自动注入公司标准文件头、版权声明。
Plugin 的文件结构
~/.claude/plugins/
└── my-plugin/
    ├── index.js      # 插件入口文件（必须）
    └── package.json  # 依赖声明（可选）

一个 Plugin 的 index.js 需要导出一个固定格式的对象，包含：插件元信息 + 工具函数列表：

// ~/.claude/plugins/company-tools/index.js

module.exports = {
  // ① 插件基本信息
  name: "company-tools",
  description: "公司内部工具集成",
  version: "1.0.0",

  // ② 工具函数列表（Claude 可以调用的函数）
  tools: [
    {
      // 工具名称（Claude 调用时使用）
      name: "get_jira_issue",

      // 工具描述（Claude 根据这个决定什么时候调用它）
      description: "根据 Issue ID 获取 Jira 任务的详细信息，包括标题、描述、状态、优先级和负责人",

      // 输入参数的 JSON Schema 定义
      inputSchema: {
        type: "object",
        properties: {
          issueId: {
            type: "string",
            description: "Jira Issue ID，格式如 PROJ-123"
          },
          includeComments: {
            type: "boolean",
            description: "是否包含评论列表，默认 false"
          }
        },
        required: ["issueId"]
      },

      // 实际执行函数
      async handler({ issueId, includeComments = false }) {
        const baseUrl = process.env.JIRA_URL;
        const token = process.env.JIRA_TOKEN;

        const response = await fetch(
          `${baseUrl}/rest/api/3/issue/${issueId}`,
          {
            headers: {
              "Authorization": `Bearer ${token}`,
              "Accept": "application/json"
            }
          }
        );

        if (!response.ok) {
          throw new Error(`Jira API 错误: ${response.status} ${response.statusText}`);
        }

        const data = await response.json();

        return {
          id: data.key,
          title: data.fields.summary,
          description: data.fields.description?.content?.[0]?.content?.[0]?.text || "无描述",
          status: data.fields.status.name,
          priority: data.fields.priority?.name || "未设置",
          assignee: data.fields.assignee?.displayName || "未分配",
          // 如果需要评论，再发一个请求
          comments: includeComments ? await getComments(issueId, baseUrl, token) : []
        };
      }
    },

    // 可以有多个工具函数
    {
      name: "create_deploy_ticket",
      description: "创建标准的部署申请单，自动填入必要字段",
      inputSchema: {
        type: "object",
        properties: {
          serviceName: { type: "string", description: "要部署的服务名" },
          version: { type: "string", description: "版本号，如 v1.2.3" },
          environment: { type: "string", enum: ["staging", "production"], description: "目标环境" }
        },
        required: ["serviceName", "version", "environment"]
      },
      async handler({ serviceName, version, environment }) {
        // ... 调用内部 API 创建工单
      }
    }
  ]
};

    "./.claude/plugins/project-specific"  // 项目级插件
  ]

"查一下 Jira PROJ-456 的需求描述，
然后根据需求生成对应的后端接口代码，
代码写完后帮我创建一个部署到 staging 的工单"
3 个常见误区
!
误区一：该用 MCP 的场景用了 Plugin
场景：想让 Claude 连接 PostgreSQL 数据库，选择自己写 Plugin。
问题：官方已有 server-postgres MCP Server，功能完整、安全性经过验证。自己写 Plugin 费时费力还容易踩坑。
原则：有官方 MCP Server 的优先用 MCP；只有公司内网系统、特殊业务逻辑才需要自己写 Plugin。
!
误区二：description 写得太模糊
问题：工具 description 写"处理 Jira"，Claude 不知道什么情况下该调用它，也不知道它能做什么。
正确：description 要写清楚"什么时候用"+"能返回什么信息"。Claude 根据 description 决定是否调用你的工具，写得越清楚，Claude 调用时机越准确。
!
误区三：handler 里没有错误处理
问题：handler 函数直接 await fetch，如果网络超时或 API 返回错误，会抛出未处理的异常，导致 Claude 工作流中断。
正确：handler 里必须有 try/catch，失败时返回一个结构化的错误对象（而不是抛出异常），让 Claude 知道发生了什么并做出合理响应。
📌 本模块核心要点
  • Plugin 改变 Claude Code 本身的行为；MCP 给 Claude 接外部工具 — 两者解决不同问题
  • Plugin 用 JavaScript 编写，导出 tools 数组，每个工具包含 name + description + inputSchema + handler
  • description 要写清楚"什么时候用"和"能返回什么"，Claude 靠它决定何时调用你的工具
  • 有官方 MCP Server 的优先用 MCP，Plugin 用于公司内网系统和特殊业务逻辑
  • handler 里必须做错误处理，返回结构化结果，不要直接抛异常

后台任务

不要让 Claude 的任务堵住你的工作。耗时任务交给后台，你去做别的事，完成了自动通知你。

⏱ 预计 45 分钟
🔴 难度：高级
同步等待的代价

你让 Claude 做一件需要 20 分钟的事——比如"扫描整个项目，为所有函数生成单元测试"。

默认情况下，你只能盯着屏幕等。Claude 在工作，你什么都不能做，窗口一关任务就断。20 分钟，就这么浪费了。

⚠️ 这是同步阻塞模式的问题。Claude 执行 = 你等待。对于短任务（几秒）这没问题，但对于长任务（几分钟甚至几十分钟），同步等待是极大的浪费。

后台任务解决的正是这个问题：任务提交后立刻返回，Claude 在后台自主执行，你去干别的，完成后收到通知。

类比：你已经在用这个模式了
📧
发邮件，不等回复
你发完邮件不会盯着收件箱等，你去做别的事，有回复了手机会响。Claude 后台任务就是这个模式：提交任务 → 去忙别的 → 完成了通知你。
🖨️
打印机后台打印
你点击"打印"后不会站在打印机旁等，你继续工作，打印完了打印机会提示。后台任务就是把 Claude 变成这样的"打印机"。
☁️
云端异步处理
上传视频到平台，不是同步等转码结束，而是提交 → 平台异步处理 → 处理完推送通知。Claude 后台任务的机制完全一样。
后台任务的完整工作流
sequenceDiagram participant U as 你 participant CC as Claude Code participant BG as 后台进程 participant H as Stop Hook（通知） U->>CC: 提交后台任务："为整个项目生成测试" CC->>BG: 创建后台进程，任务 ID: task_001 CC-->>U: 立即返回："任务已启动，可以关闭窗口" Note over U: 你去做别的事情 ☕ BG->>BG: 自主分析项目结构 BG->>BG: 逐模块生成测试文件 BG->>BG: 运行测试验证通过率 BG->>BG: 生成覆盖率报告 BG->>H: 任务完成，触发 Stop Hook H-->>U: 推送通知："测试生成完成，覆盖率 76%，3个模块失败"
两种启动方式

方式一：CLI 参数启动

claude --background "扫描整个项目，为覆盖率低于 50% 的模块生成单元测试，
完成后生成覆盖率对比报告"

# 可以同时启动多个后台任务
claude --background "重构 src/utils/ 目录，统一错误处理"
claude --background "为 src/api/ 下所有接口生成 API 文档"
# 两个任务并行执行！

方式二：对话中明确声明

# 在对话里告诉 Claude 以后台模式执行
请以后台任务模式完成以下工作，不需要我一直等待：

任务：
1. 扫描所有 TypeScript 文件，找出缺少类型注解的函数参数
2. 自动补全类型注解（基于上下文推断）
3. 运行 tsc --noEmit 确认无类型错误
4. 统计修改了多少个文件、添加了多少处注解
5. 如果某个文件类型推断不确定，标记出来请我人工确认

完成后，输出修改摘要报告。不需要实时输出进度。

哪些任务适合放后台？
任务类型预计时长是否适合后台理由
修复一个 Bug< 30 秒❌ 不适合同步更快，后台有额外启动开销
批量生成测试（10+ 文件）5-20 分钟✅ 非常适合耗时长，你可以去做别的
全项目重构10-60 分钟✅ 强烈推荐必须后台，否则你会等到崩溃
批量文档生成5-30 分钟✅ 非常适合不需要实时监控进度
代码解释（需要交互）即时❌ 不适合需要实时问答，后台无意义
安全扫描全项目10-30 分钟✅ 非常适合适合离开电脑时在后台跑
3 个常见误区
!
误区一：任务描述不够完整就扔到后台
问题：任务描述模糊（"重构一下代码"），Claude 在后台执行到一半遇到歧义，只能瞎猜或者停下来，你回来发现结果一团糟。
正确：后台任务的描述要比普通任务更详细——明确范围、明确约束、明确输出格式。因为中途你不在旁边，Claude 没有机会问你。
!
误区二：后台任务跑完了不知道
问题：没配置 Stop Hook，任务跑完了没有通知，你一直在等，其实早就完成了。
正确：后台任务必须配套 Stop Hook 通知。哪怕只是一个简单的系统通知或钉钉消息，也能极大提升使用体验。
!
误区三：并行启动太多后台任务
问题：同时启动 10 个后台任务，它们同时写文件，互相产生冲突，文件内容混乱。
正确：并行任务需要确保操作的文件/目录不重叠。建议一次并行不超过 3-4 个，且每个任务有明确独立的操作范围。
📌 本模块核心要点
  • 后台任务 = 提交 → 立刻返回 → Claude 自主执行 → 完成通知，解放你的等待时间
  • 适合：批量生成测试、全项目重构、文档生成、安全扫描等 5 分钟以上的任务
  • 后台任务描述必须更详细，因为中途没有机会追问
  • 必须配置 Stop Hook 通知，否则任务完成了你不知道
  • 并行任务要确保操作范围不重叠，避免文件写入冲突

三种使用模式，各有适用场景

Claude Code CLI 不只是"打开聊天框"这一种用法，理解三种模式能让你在不同场景选对工具：

💬
交互模式（默认）
直接输入 claude，进入对话界面，支持多轮对话、实时交互。
适合：日常开发、探索性任务、需要多轮确认的工作
⚡

输入 claude "任务描述"，执行完立刻退出，输出结果到终端。
适合：脚本调用、CI/CD 流程、批量处理
🔗

通过 stdin 接收输入，配合 Unix 管道使用。
适合：处理命令输出、与其他工具组合使用
核心参数速查
参数说明实际用法示例
claude 启动交互式会话 claude（在项目目录下运行，自动读取 CLAUDE.md）
claude "任务" 非交互式，执行单条任务后退出 claude "为 src/user.ts 生成单元测试"

权限控制规则详解

权限控制是 settings.json 最重要的安全配置，特别是在 CI/CD 或多人团队中：

规则写法含义推荐场景
Bash(*)允许所有 Bash 命令本地开发，完全信任环境
Bash(git *)只允许 git 开头的命令只需要 Git 操作的场景
Bash(npm run *)只允许 npm run xxx 命令限制只能跑预定义脚本
Write(src/*)只允许写 src/ 下的文件防止 Claude 修改配置文件
Read(src/*)只允许读 src/ 下的文件保护敏感目录（如 .env）
Write允许写入任意文件通用开发场景
💡 安全建议：在 CI/CD 环境里，建议用最小权限原则——只 allow 任务需要的操作，其余全部隐式 deny。比如文档生成任务只需要 Read + Write(docs/*)，不需要任何 Bash 权限。
📌 本模块核心要点
  • 三种模式：交互模式（日常）/ 单次执行（脚本/CI）/ 管道模式（与 Unix 工具组合）
  • 管道模式 -p 让 Claude 成为 Shell 工具链的一员，配合 git diff、日志分析威力巨大
  • CI/CD 中配合 --no-stream --output-format json 输出结构化结果，方便程序解析
  • settings.json 的权限控制是安全基础，CI 环境下用最小权限原则
  • API Key 只通过环境变量传入，永远不写进代码或配置文件

综合工作流

前面 9 个模块是各自独立的能力，这个模块告诉你怎么把它们组合起来，打造真正属于你的 AI 开发工作流。

⏱ 预计 90 分钟
🔴 难度：综合
一个核心思路：能力叠加，不是随机组合

学完前面 9 个模块，你可能会有一种冲动：把所有能力都堆进去！CLAUDE.md + Skills + Hooks + MCP + Plugin + 子代理 + 后台任务……

但真正有效的工作流，是针对具体场景，选择最合适的几个能力组合，而不是把所有东西都堆在一起。

💡 组合原则：① 找到你工作中最痛的重复性环节；② 识别哪些模块能解决这个环节；③ 最小化组合，验证效果后再扩展。不要一次性把所有模块都用上。
